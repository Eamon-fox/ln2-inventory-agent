# 界面展示层

模块 ID：`gui_presentation`

本页默认按软约束和背景说明理解。真正硬约束只以 `docs/01-系统架构总览.md`、`docs/02-模块地图.md`、`docs/03-共享瓶颈点.md` 的机器可读契约为准。

## 模块定位

本模块负责所有直接面向用户的 Qt 展示与交互组件。它的职责是展示状态、收集输入、触发稳定入口，不负责定义库存规则、导入策略或工具契约。

如果一个改动的目标是“让界面显示正确、交互顺手、输入可控”，它大概率属于本模块。

## 拥有路径

- `app_gui/ui/`

## 主职责

- 面板、对话框、表格、列表、控件布局
- 面板内局部交互状态
- 用户输入约束与前端提示
- 面板级信号定义与视觉反馈

## 允许依赖

- `app_gui/application/` 暴露的稳定流程入口
- `app_gui/plan_*.py` 的结果模型或只读协作能力
- `lib/` 中纯展示辅助能力，例如位置格式化、只读解析或文本格式化

## 禁止依赖

- `lib.tool_api_write*`
- `lib.tool_api_write_validation`
- `agent/`
- 把库存写入规则、审计规则、导入规则直接写进 UI 组件

## 与其他模块的边界

### 与 `gui_application` 的边界

- 展示层负责“显示什么按钮、输入什么值、触发什么稳定调用”
- 应用层负责“这次操作要怎么编排、怎么跨面板同步、怎么发布事件”

### 与 `inventory_core` 的边界

- 展示层可以显示来自核心层的数据
- 展示层不应决定核心字段规则、工具契约、写入顺序或审计策略
- 与位置索引相关的显示面（例如 Overview、Plan、打印预览、导出前展示）默认复用 `lib.position_fmt` 的显示 helper，不要在 UI 里手拼 `box:position` 或直接暴露内部整数位置。

### 与 `agent_runtime` 的边界

- 展示层只能展示 agent 的结果和事件
- 不要把 agent 的推理、恢复策略、工具选择逻辑塞回 UI
- AI 面板的流式展示按 agent 事件的 `meta.channel` 做视觉分流：`thought` 只进入思考面板，`answer` 只进入最终回答区域。
- `thought` 是展示层局部状态，不是最终回答正文；UI 不应自行解释 reasoning、tool results 或历史拼接语义。
- 思考面板仅在 `thought` chunk 活跃输出期间临时显示；一旦进入 `answer`、工具边界或流结束即自动隐藏，不保留历史 reasoning 展开入口。

## 盒展示语义

- UI 可以显示盒子的可选标签，但必须继续把数字盒号作为真实定位键。
- 概览、打印、导出等展示面可以把标签作为补充信息展示，不要让标签单独承担身份语义。
- 与 `meta.box_layout.box_tags` 相关的用户文案可以继续使用“Tag / 标签”，但不要暗示底层盒身份已改成字符串。

## Overview 交互语义

- `OverviewPanel` 网格中的占用格，单击默认用于预填到操作面板；不要把普通浏览、点选或短距离拖动直接判定为移动操作。
- 从概览网格发起移动，默认采用”长按后拖拽”这类防误触手势，再通过 `plan_items_requested` 进入计划区；展示层不应在拖拽完成时直接执行写入。
- 表格视图中已暂存的计划项可以通过 `plan_item_removal_requested` 信号请求移除；该信号连接到 `OperationsPanel.remove_plan_items_by_payload`，由操作面板侧统一操作 `PlanStore`。
- 长按时长、拖拽距离这类参数属于展示层内部交互策略，默认按软约束记录在本模块文档与对应 GUI 测试中，不上提到顶层硬契约，除非它已经变成跨模块共同依赖的稳定边界。
- 空格上若存在 `plan_store` 的待执行 add 操作（`action == "add"` 且 payload 覆盖该 position），概览网格应在格内把 `display_key` 的待填值以占位样式显示出来，而不是继续显示位置号；`plan_model.apply_operation_markers_to_grid` 的打印视图需表达同一语义。锁测试位于 `tests/unit/test_overview_panel_add_preview.py` 与 `tests/integration/plan/test_plan_model.py`（`test_apply_operation_markers_attaches_preview_label_for_add` 等）。

### 表视图增量渲染契约

- `app_gui/ui/overview_panel_table._render_table_rows` 默认走「按行签名对比」的增量路径：同一行的 `row_kind / box / position / record_id / color_value / row_confirmed / row_locked / 展示列值` 未变时，跳过 `setItem`。
- 当行数变化、列集合变化或列类型映射变化时回退到整表 `setRowCount + 重建`；这是唯一允许重建的触发点，其他刷新路径不得重新调用 `setRowCount(0)`。
- `_refresh_table_entry_row_visual` 单行重绘后，必须同步更新缓存签名，避免下一次批量渲染误判「已同步」。
- 该契约的锁测试位于 `tests/integration/gui/test_gui_panels_data_views.OverviewTableViewTests::test_render_table_rows_uses_set_row_count_batch_path` 与同类 `test_render_table_rows_only_rewrites_changed_rows`；若要改变签名字段，请同步更新这两处与本节文字。

## 共享瓶颈点

以下文件虽然经常与展示层任务相关，但不属于本模块可自由改动的内部文件：

- `app_gui/main.py`
- `app_gui/main_window_flows.py`
- `app_gui/tool_bridge.py`
- `app_gui/i18n/translations/en.json`
- `app_gui/i18n/translations/zh-CN.json`

一旦任务需要改这些文件，就按跨模块任务处理。

## 适合的任务切分方式

适合并行拆分为：

- 单个面板内的布局重构
- 某个对话框或表格的交互细化
- 某个 UI 子模块的视觉与输入限制调整

不适合并行拆分为：

- 多个任务同时改同一个面板的公共方法签名
- 一边改 UI，一边改主窗口 wiring

## 典型反模式

- 为了“先能跑”，在 `app_gui/ui/` 里直接写工具调用编排
- 让 UI 组件直接导入底层写校验模块
- 依赖另一个面板的私有字段或私有方法
- 在 UI 层复制业务规则，导致 GUI 与 agent 行为不一致

## 最小测试责任

- `pytest -q tests/integration/gui`

如果只改某个具体面板，至少补跑对应的 GUI 集成测试文件。
