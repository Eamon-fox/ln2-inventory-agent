# 发布与安装

模块 ID：`release_packaging`

本页默认按软约束和背景说明理解。真正硬约束只以 `docs/01-系统架构总览.md`、`docs/02-模块地图.md`、`docs/03-共享瓶颈点.md` 的机器可读契约为准。

## 模块定位

本模块负责把应用打成可交付产物，并维护版本、安装器、发布脚本和网站同步逻辑。它解决的是“如何交付”，不负责定义业务规则。

如果一个改动的目标是“更新版本、修改打包脚本、调整安装器或发布流程”，它属于本模块。

## 拥有路径

- `installer/`
- `scripts/`
- `ln2_inventory.spec`
- `ln2_inventory.mac.spec`
- `latest.json`
- `CHANGELOG.md`

## 主职责

- Windows 与 macOS 打包
- 安装器脚本
- 发布版本号与下载地址同步
- 发布说明与历史版本维护
- 发布脚本与网站同步脚本

## 当前主要协作面（软约束）

以下内容视为本模块的稳定入口或稳定资产：

- `installer/windows/`
- `installer/mac/`
- `scripts/README.md`
- `latest.json`
- `CHANGELOG.md`

## 发版执行手册

如果任务不是“改脚本”而是“正式发版”，先补读：

- `docs/runbooks/25-发布版本手册.md`

该手册负责回答：

- 哪个文件是版本真相源
- 哪些产物必须一起同步
- Windows 构建、OSS 上传、网站同步和发布验收的完整顺序
- 常见失败点与回滚点

## 允许依赖

- 读取版本号、构建产物路径、发布元数据

## 禁止依赖

- 把库存规则、导入规则或 GUI 业务逻辑写进打包与发布脚本
- 用发布脚本去修补业务数据问题

## 与其他模块的边界

### 与业务模块的边界

- 发布与安装模块只消费构建结果和版本信息
- 业务模块不应依赖安装器脚本才能正常工作

### 与翻译和文案的边界

- 发布相关文案可以更新
- 但通用 GUI 文案和业务翻译仍归应用/UI 侧维护

## 共享风险点

本模块虽然通常能和业务模块并行，但以下场景要特别小心：

- 更新版本号时，多个任务同时动 `latest.json`
- 更新发布说明时，多个任务同时改 `CHANGELOG.md`
- 网站同步脚本与安装包文件名规则同时变化

## 适合的任务切分方式

适合并行拆分为：

- 一个任务改安装器
- 一个任务改版本同步脚本
- 一个任务整理发布说明

不适合并行拆分为：

- 多个任务同时修改版本号来源
- 一边改安装包命名规则，一边改网站同步逻辑但没有统一收口

## 典型反模式

- 在打包脚本里偷偷修正业务配置
- 用发布脚本承担迁移逻辑
- 发布资产命名与 `latest.json` 不一致
- 文档写一套、脚本实际做另一套

## 最小测试责任

- 至少执行与本次改动直接相关的脚本自检或构建验证

如果改了版本同步或安装器脚本，应在说明中写清楚验证方式和受影响产物。
