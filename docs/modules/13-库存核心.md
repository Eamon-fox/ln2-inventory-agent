# 库存核心

模块 ID：`inventory_core`

本页默认按软约束和背景说明理解。真正硬约束只以 `docs/01-系统架构总览.md`、`docs/02-模块地图.md`、`docs/03-共享瓶颈点.md` 的机器可读契约为准。

## 模块定位

本模块是系统的统一业务内核。它定义库存工具入口、字段与校验规则、工具契约、注册关系、YAML 业务处理与库存真相模型。

如果一个改动的目标是“改变系统如何理解库存数据或如何执行库存操作”，它属于本模块。

## 拥有路径

- `lib/`

以下专题路径虽然也在 `lib/` 下，但语义归属于导入迁移模块：

- `lib/import_*`
- `lib/migrate_*`

## 主职责

- `lib/tool_api.py` 统一工具入口
- `lib/tool_registry.py` 工具注册、描述符定义与工具契约真相源
- 库存校验、字段规则、路径与 YAML 业务处理
- 领域命令与领域事件
- 自定义字段、搜索、统计、导出等通用业务服务

## 当前主要协作面（软约束）

以下文件和模块视为库存核心稳定入口：

- `lib/tool_api.py`
- `lib/tool_registry.py`
- `lib/domain/commands.py`
- `lib/domain/events.py`
- Shared Overview table query truth should live below the chokepoints (for example `lib/overview_table_query.py`) and be exposed outward through `tool_filter_records` rather than being duplicated in GUI or agent code.
- 位置显示/输入的真相源默认收口到 `lib.position_fmt`；GUI、导出、打印等上层展示面应复用它，而不是各自再拼一套位置格式化逻辑。
- Box-layout 变更的 request normalization、值归一化与 dry-run preview 真相应收口在核心层下沉模块中，而不是散落在 GUI flow、Agent handler 和多个写实现里。

`lib/tool_registry.py` 中与写工具相关的稳定职责应限定为：

- 公共契约元数据
- write/migration/public 分类
- 显式 `write_api_attr`
- agent dispatch / GUI bridge 的绑定元数据

与 LLM 运行时体验相关的提示、状态文案、hooks、输入守卫不属于本模块真相源。

历史 YAML 的兼容策略在本模块中统一收口为“首次打开自动迁移”：当检测到 legacy 字段或旧结构别名时，先备份、再 canonicalize、再记录审计事件；迁移完成后，日常运行链路只处理 canonical 数据。

`data_root` 迁移后的内容级修正也应收口在本模块：当受管数据集被整体复制到新的数据根目录后，库存核心负责同步修正仍会参与运行时严格路径校验的持久化字段，例如 YAML 中的 `meta.instance_origin_path`，以及审计 JSONL 顶层的 `yaml_path` / `backup_path`。这类规则不应散落在 GUI、设置对话框或更底层的通用存储复制逻辑中；历史说明型 `details.*path` 字段默认保留原值，除非后续它们也进入运行时真相链路。

新增工具、删除工具、修改统一返回格式、调整事件契约时，都应视为架构级改动。

## 校验错误输出契约（软约束）

库存校验链路 `validate_inventory` / `validate_record` 的错误输出既要保持历史的人类可读字符串，也要承载结构化元数据，供 GUI 等上层定位问题。

- 错误项类型：`lib.validation_primitives.ValidationMessage`，它继承自 `str`，因此所有把校验结果当字符串消费的旧代码（拼接、日志、测试文本断言）不受影响。
- 每个 `ValidationMessage` 通过 `.detail` 字典（以及 `extract_error_details(errors)` 辅助函数）对外暴露：
  - `rule`：机器可读的规则名，如 `missing_required` / `invalid_id` / `box_out_of_range` / `position_out_of_range` / `position_null_without_takeout` / `invalid_date` / `date_in_future` / `duplicate_id` / `position_conflict`。
  - `field`：涉及的字段名（可空）。
  - `value` / `expected`：出错值与期望值或允许集合（可空）。
  - `record_id` / `record_index` / `box` / `position`：定位信息（可空）。
  - `message`：和 `str(err)` 等价的人类可读文本兜底。
- 统一写入链路通过 `lib.tool_api_support._validate_data_or_error` 将结构化详情以 `errors_detail: list[dict]` 透出到工具返回值；`errors: list[str]` 字段继续保留以兼容历史消费者。
- 允许新增规则名或字段键，但不得悄悄改掉已有规则名的语义；规则重命名视为契约变更，应同步更新本节与相关测试。

## 盒身份语义

- `meta.box_layout.box_numbers` 是盒级别的稳定数字身份，不是展示字段。
- `inventory[].box` 只能引用这个数字身份；库存索引、映射和导出定位都继续以它为准。
- `meta.box_layout.box_tags` 只是按盒号挂载的可选标签；标签可以是盒子的名字，但不代表它承担身份职责。
- `set_box_tag` 这类写操作默认只修改标签，不修改盒身份，也不引入字符串型盒主键。
- `meta.box_layout.indexing` 只定义位置的显示/输入模式；切换它时不得把 `inventory[].position` 从整数改写成字符串。

## 允许依赖

- 基础设施能力
- 通用 Python 标准库与纯业务服务

## 禁止依赖

- `PySide6`
- GUI 状态对象
- GUI 组件或窗口类
- 依赖 `agent/` 的推理策略

## 与其他模块的边界

### 与 `agent_runtime` 的边界

- 核心层给出工具真相与规则
- agent 只负责调度，不得复制规则
- `manage_boxes` 这类 box-layout 写入工具的参数归一化与合法性判断，默认不应由 agent 自己再维护一份。

### 与 `gui_application` 的边界

- 应用层组织调用
- 核心层保持规则统一，不关心具体面板协作
- `set_box_tag`、`set_box_layout_indexing`、`manage_boxes` 的写前校验和 preview 语义应由核心层统一定义；GUI 可以发起 dry-run，但不应自己再实现规则判断。

### 与 `migration_import` 的边界

- 核心层提供通用库存模型与验证能力
- 导入迁移模块负责外部输入的映射、预处理与导入流程

## 共享瓶颈点

本模块内部有 2 个最高冲突文件：

- `lib/tool_api.py`
- `lib/tool_registry.py`

这些文件应尽量薄化，承载统一入口与注册真相源，不要继续沉积模块私有实现。

## 推荐改动策略

1. 先把具体规则实现下沉到更内层模块或服务函数
2. 最后再改 `tool_api.py` 接线
3. 需要注册新工具时，最后一步再改 `tool_registry.py`
4. 历史数据兼容统一通过升级入口处理，不要把 legacy 分支继续散落到 steady-state runtime

## 适合的任务切分方式

适合并行拆分为：

- 一个任务改底层规则实现
- 一个任务改工具注册与统一入口接线
- 一个任务补测试与升级入口覆盖

不适合并行拆分为：

- 多个任务同时改 `tool_registry.py`
- 一边改规则，一边多个调用侧同时各自猜测最终契约

## 典型反模式

- 把 UI 或 agent 的特殊需求直接写死在核心层
- 让统一入口越来越厚，变成巨型脚本
- 在多个地方重复维护工具契约
- 在 GUI flow、Agent handler、写实现里各自维护一份 box-layout 请求校验
- 让 `tool_registry.py` 同时承担公共契约和 agent 运行时特例，重新变回跨模块巨型配置文件
- 让兼容层重新成为真正的真相源

## 最小测试责任

- `pytest -q tests/integration/inventory tests/unit`

若动到工具契约、统一入口或领域事件，建议额外补跑 `tests/contract` 中相关用例。
