# 智能体运行时

模块 ID：`agent_runtime`

本页默认按软约束和背景说明理解。真正硬约束只以 `docs/01-系统架构总览.md`、`docs/02-模块地图.md`、`docs/03-共享瓶颈点.md` 的机器可读契约为准。

## 模块定位

本模块负责 AI 智能体的推理循环、工具调度、工具输入校验与恢复策略。它的目标不是持有一套独立业务真相，而是安全、高效地驱动统一 Tool API。

如果一个改动的目标是“让 agent 更会调用工具、更稳地恢复错误、更清楚地管理上下文”，它属于本模块。

## 拥有路径

- `agent/`

## 主职责

- ReAct 循环
- 历史压缩与上下文一致性维护
- 外部上下文检查点摘要与恢复
- LLM 调用与模型适配
- 工具调用调度
- 工具 hooks、校验与阶段化拦截
- Agent 可见命令行工具的会话状态（`shell` / `current_workdir`）
- 用户问题与确认流控制
- 运行时特例注册：状态文案、输入守卫、schema 增强、错误提示、before/after hooks

## 当前主要协作面（软约束）

以下入口视为本模块的稳定协作面：

- `ReactAgent`
- `AgentToolRunner.list_tools`
- `AgentToolRunner.run`
- `AgentToolRunner` 的调度面与 staging 面
- Agent staging should write transient plan validation state through `PlanStore.apply_validation_state()` / `update_validation_statuses()`, not by creating private `_validation_*` plan item fields.

调用方应依赖这些入口，而不是依赖某个 handler 的内部实现细节。

## 允许依赖

- `inventory_core`
- `migration_import`

## 禁止依赖

- `PySide6`
- GUI 组件或窗口对象
- 复制 `inventory_core` 的业务规则、工具契约或字段真相
- 直接绕过统一入口写 YAML

## 与其他模块的边界

### 与 `inventory_core` 的边界

- 智能体运行时负责“怎么调用”
- 库存核心负责“什么是合法规则”
- `lib/tool_registry.py` 只提供公共契约、分类、调度绑定与显式写入口
- 与 LLM/交互体验强相关的运行时特例，应留在 `agent/` 内部注册表，不要反向塞回共享注册表

### 与 `gui_application` 的边界

- GUI 负责展示与人工确认
- agent 负责推理、工具编排与阶段化输出
- GUI 会话层可以持有 `summary_state` 这类 opaque session 状态，但不得自己解释或重写摘要内容
- GUI 会话层可以持有 shell session 状态，但只负责保存、回传和在 new chat 时清空；`current_workdir` 的更新规则归 agent runtime 所有

### 命令行工具边界

Agent 只应看到一个命令行工具：`shell`。

- `shell` 默认使用当前 AI 会话的 `current_workdir`
- `workdir` 参数只接受 repo-relative 路径
- `engine` 仅作为可选底层执行选择：`auto` / `bash` / `powershell`
- `bash` 与 `powershell` 可以作为内部执行引擎存在，但不应作为 Agent 可见工具暴露

### 与 `migration_import` 的边界

- agent 可承接迁移工作流的执行
- 迁移模块负责 workspace、模板、验收与导入规则

## 共享瓶颈点

以下文件虽然不在 `agent/` 下，但 agent 任务经常需要碰到，因此默认按共享瓶颈点处理：

- `lib/tool_registry.py`
- `app_gui/tool_bridge.py`

新增工具、调整工具契约、修改 GUI 侧桥接语义时，不要假设这是纯 agent 任务。

## 外部上下文检查点

当主会话即将接近模型上下文预算时，默认不再沿用“按固定消息条数压缩旧历史”的主路径，而是触发一次独立的外部检查点摘要：

- 摘要调用与主 agent 使用同一个 provider/model
- 摘要调用在 fresh context 中执行，不复用主会话消息
- 摘要调用不启用 tools
- 摘要 prompt 与 resume prompt 都是固定提示词，不受用户 custom prompt 覆盖
- 摘要结果以 `summary_state` 形式回流，作为 agent runtime 拥有的 session 内部状态
- 主会话恢复时，agent 必须同时看到固定 resume prompt 与摘要内容，再继续运行

`summary_state` 应被视为 runtime 的不透明状态对象。GUI 只负责在当前 AI 会话内保存、回传和在 new chat 时清空，不负责自行生成或修改摘要语义。

## 适合的任务切分方式

适合并行拆分为：

- 一个任务改历史与提示词策略
- 一个任务改工具 hooks 或错误恢复
- 一个任务单独处理工具注册或桥接接线

不适合并行拆分为：

- 多个任务同时改工具注册表
- 一边改 agent 的工具参数假设，一边不更新核心层契约

## 典型反模式

- 在 prompt 或 runtime 里偷偷定义新字段别名，但核心层并不认
- 为某个工具临时补一段“仅 agent 知道”的业务规则
- 为历史文件继续在 agent runtime 里保留长期 legacy 参数兼容
- 为了加一个工具提示/状态文案/输入守卫，同时改 `tool_runner_guidance.py`、`tool_status_formatter.py`、`tool_hooks.py`、`tool_runner_validation.py` 多处散落表
- 让 agent 直接依赖 GUI 私有实现
- 把业务数据修复逻辑写死在 prompt 文本里
