# 界面应用层

模块 ID：`gui_application`

本页默认按软约束和背景说明理解。真正硬约束只以 `docs/01-系统架构总览.md`、`docs/02-模块地图.md`、`docs/03-共享瓶颈点.md` 的机器可读契约为准。

## 模块定位

本模块负责 GUI 场景下的流程编排与跨组件协作。它承接展示层输入，把局部交互升级成“可执行的应用流程”，并把核心层能力以 GUI 需要的方式组织起来。

如果一个改动的目标是“多个面板如何协作、计划如何执行、数据集如何切换、主窗口如何装配”，它属于本模块。

## 拥有路径

- `app_gui/application/`
- `app_gui/plan_*.py`

## 共享但由本模块主导语义的文件

以下文件不属于本模块独占路径，但其语义主导权在本模块：

- `app_gui/main.py`
- `app_gui/main_window_flows.py`
- `app_gui/tool_bridge.py`

它们同时也是共享瓶颈点，默认不能多人并行修改。

## 主职责

- GUI 用例与流程编排
- 数据集切换、迁移模式切换、计划执行协作
- 事件总线与事件消费
- 主窗口级状态同步
- GUI 到统一 Tool API 的桥接
- 嵌入 GUI 进程的本地开放 API（只读查询/校验、受管数据集会话切换与 GUI handoff）

## 当前主要协作面（软约束）

以下入口视为 GUI 应用层稳定入口：

- `DatasetUseCase.switch_dataset`
- `MigrationModeUseCase.set_mode`
- `PlanExecutionUseCase.report_operation_completed`
- `PlanRunUseCase.execute`
- `DatasetLifecycleUseCase.resolve_startup_yaml_path` / `create_dataset` / `rename_dataset` / `delete_dataset`
- `SettingsDialogSubmission` ?? Settings ????????????
- `HelpDialog` 承载帮助、关于、反馈入口与检查更新，不参与 Settings 保存契约
- `GuiToolBridge` 对 GUI 暴露的方法集合
- Box-layout 相关流程应优先复用库存核心的 request normalization / dry-run preview，而不是在 GUI 自己再定义一套写前校验。
- Overview table mode should reuse `GuiToolBridge.filter_records` for field filters, in-table search, and sorting instead of reimplementing table query semantics inside the panel.
- Staged plan validation state should be consumed through `PlanStore.validation_state()` / `PlanStore.validation_status()` rather than by reading transient `_validation_*` item keys directly.

新增、删除或重命名这些入口时，应同步更新架构文档。

## 允许依赖

- `gui_presentation`
- `inventory_core`
- `migration_import`

## 禁止依赖

- 依赖 UI 组件的私有字段、私有子控件结构
- 在本模块复制库存写入规则或工具契约
- 把应用协调逻辑散落回展示层

## 与其他模块的边界

### 与 `gui_presentation` 的边界

- 展示层负责局部交互
- 应用层负责跨面板、跨流程、跨状态同步

### 与 `inventory_core` 的边界

- 应用层负责组织调用
- 核心层负责定义规则、校验与统一返回面
- box-layout 变更的请求归一化、合法性判断与 dry-run 预演归核心层真相源所有；应用层只负责输入采集、确认流与结果分发。

### 与 `migration_import` 的边界

- 应用层负责 GUI handoff 和数据集切换
- 导入迁移模块负责迁移工作区、导入策略与迁移验收

## 共享瓶颈点中的约束

### `app_gui/main.py`

- 只保留装配、初始化与顶层信号连接
- 不要把具体业务规则继续塞进主窗口类
- 启动时须先取得 `app_gui.single_instance.SingleInstanceLock`（位于 `get_user_config_dir()/instance.lock` 的 OS 级文件锁）；取锁失败直接弹 `main.singleInstanceTitle/Message` 并退出 0。锁必须跨 `_bootstrap_data_root` 与 `app.exec()` 全程持有，仅在进程退出前 `release()`。锁测试由 `tests/unit/test_single_instance_lock.py` 约束。

### macOS 更新 UX 契约

SnowFox 的 macOS 分发走 `.pkg` + Installer.app（见 `installer/mac/build_pkg.sh`，`--install-location /Applications`）。Installer.app 安装的 payload 不会被打上 `com.apple.quarantine` xattr，因此在同一 bundle id 下，自动重启 `/Applications/SnowFox.app` 不会再触发 Gatekeeper 二次放行。更新流程按下列契约工作，与 Windows 分支对齐：

- `StartupFlow.start_automatic_update` 在 `platform_key == "macos"` 且 `auto_update=True` 时，必须先通过 `_confirm_macos_update()` 显示预安装确认（`main.macosUpdatePreInstallMessage`），用户取消则直接终止，不启动 `AutoUpdater`。
- `platform_key == "macos"` 且 `auto_update=False` 的手动下载路径保持不变，仍使用 `main.updateManualMessageMacos`。
- macOS 自动更新完成后，`_on_complete` 必须用 `main.macosUpdatePostInstallMessage` 取代 `AutoUpdater` 默认消息，并且**必须**通过 `QTimer.singleShot(delay, app.quit)` 让老进程自动退出（与 Windows 分支一致）。
- `auto_updater.py::_install_and_restart_macos` 的 shell 脚本**必须**包含：(1) `kill -0 {pid}` 循环等待老进程退出；(2) `open -W` 阻塞直到 Installer.app 退出；(3) `open -a "/Applications/SnowFox.app"` 拉起新版本；(4) 清理 `temp_dir`。relaunch target 固定为 `/Applications/SnowFox.app`（由 pkg `--install-location` 保证），不从 `sys.executable` 推断，避免 dev 环境误拉。
- 以上行为由 `tests/integration/gui/test_macos_updater_flow.py` 与 `tests/unit/test_auto_updater.py` 锁定。

### `app_gui/main_window_flows.py`

- 只保留窗口级流程与协作收口
- 避免继续堆积细碎规则
- 不要继续把 box-layout 写前校验、YAML 读取判断或其他核心层规则留在这里。

### `app_gui/tool_bridge.py`

- 只保留桥接、路径守卫、context 注入、失败包装
- 真正的规则应下沉到核心层或应用服务中

### 本地开放 API

- 只允许 loopback 场景下的查询、校验、受管数据集会话切换与 GUI handoff，不承担远程服务职责。
- route 暴露必须走显式 allowlist，不要从 `tool_registry` 或 `agent` 自动导出。
- route allowlist、自描述能力与请求参数面应由同一份显式 route contract 推导，避免第二真相源。
- 能力自描述与 GUI staged-plan 的只读查看，允许作为本地 Open API 的可用性增强留在应用层；但不要把它继续扩成通用本地状态服务。
- API 请求若需要操作窗口或面板状态，必须通过主线程调度收口，不要在 HTTP worker 线程里直接碰 Qt 对象。

## 适合的任务切分方式

适合并行拆分为：

- 一个任务改应用层 use case
- 一个任务改对应的 UI 触发点
- 最后单独一个任务做主窗口接线或 bridge 映射

不适合并行拆分为：

- 多个任务同时改 `main.py` 或 `tool_bridge.py`
- 一边改 plan 执行模型，一边让多个面板同时改调用语义

## 典型反模式

- 把主窗口当成“万能控制器”
- 让 bridge 直接承载业务判断
- 用 UI 私有结构代替稳定用例入口
- 在应用层复制一套核心层校验

## 最小测试责任

- `pytest -q tests/integration/plan tests/integration/gui`

如果动到 `app_gui/main.py` 或 `app_gui/tool_bridge.py`，应优先补跑与主窗口和 bridge 相关的 GUI 集成测试。
