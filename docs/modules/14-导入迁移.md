# 导入迁移

模块 ID：`migration_import`

本页默认按软约束和背景说明理解。真正硬约束只以 `docs/01-系统架构总览.md`、`docs/02-模块地图.md`、`docs/03-共享瓶颈点.md` 的机器可读契约为准。

## 模块定位

本模块负责外部数据进入系统之前的整理、迁移、预处理、验收与导入流程。它解决的是“外部材料如何安全变成系统认可的数据集”，而不是“库存日常如何操作”。

如果一个改动的目标是“导入 Excel/旧格式、迁移字段、生成中间产物、验收候选 YAML”，它属于本模块。

## 拥有路径

- `migrate/`
- `migration_assets/`
- `lib/import_*`
- `lib/migrate_*`
- `app_gui/import_journey.py`

## 主职责

- 导入工作区管理
- XLSX 预转换与标准化资产生成
- 迁移提示词、模板与清单
- 候选 YAML 验证
- 导入确认与目标数据集落地
- 导入完成后的 GUI handoff

## 当前主要协作面（软约束）

以下入口视为本模块稳定入口：

- `ImportJourneyService`
- `lib/import_acceptance.py`
- `lib/migrate_cell_line_policy.py`
- `migrate/` 工作区协议

## 允许依赖

- `inventory_core`
- `gui_application`

## 禁止依赖

- 把迁移细节塞进 `gui_presentation`
- 在迁移模块里定义独立库存真相源
- 直接修改现有 managed dataset 来“顺手比对”，除非用户明确要求

## 与其他模块的边界

### 与 `inventory_core` 的边界

- 导入迁移模块负责处理外部输入的不规范性
- 库存核心负责系统内部最终认可的模型与校验规则

### 与 `gui_application` 的边界

- GUI 应用层负责启动导入流程、接收结果、切换数据集
- 导入迁移模块负责 staging、预处理、验收与导入语义

### 与 `agent_runtime` 的边界

- agent 可执行迁移流程
- 迁移模块负责定义 workspace 协议、确认步骤与阻塞条件

## 工作区边界

迁移相关自动化默认只应在以下路径内工作：

- `migrate/inputs/`
- `migrate/normalized/`
- `migrate/output/`

这条规则的意义是隔离上下文，避免 agent 在导入流程中误读或误写现有库存数据集。

## 适合的任务切分方式

适合并行拆分为：

- 一个任务改导入前处理或标准化
- 一个任务改迁移模板与清单
- 一个任务改导入验收与最终 handoff

不适合并行拆分为：

- 多个任务同时改变迁移 workspace 协议
- 一边改导入验收，一边改最终导入目标语义但不更新文档

## 典型反模式

- 为了图快，直接跳过预处理或验收
- 让导入脚本偷偷依赖 GUI 私有逻辑
- 把迁移真相写进模板文案而不是写进代码与契约
- 混用 `migrate/` 工作区和 `inventories/` 正式数据目录
