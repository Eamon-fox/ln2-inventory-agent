# 约束模型

本文档定义本仓库的三档约束模型，用来避免“文档过少导致架构漂移”，也避免“文档过多导致实现冻结”。

## 核心原则

文档只应强约束模块之间的接缝，不应过度约束模块内部的针脚。

换句话说：

- 该约束的是边界、ownership、共享瓶颈点、少量稳定入口
- 不该约束的是 helper 命名、类内拆分、局部重构方式、尚未稳定的接口面

## 三档约束

### 1. 硬约束

硬约束必须遵守，适合进入机器可读契约，并允许 contract tests 检查。

只有以下内容应该进入硬约束：

- 分层依赖禁止项
- 模块 ownership 与共享路径
- 共享瓶颈点列表
- 数据安全或工作区边界这类高风险边界
- 极少数真正成熟且长期稳定的跨模块入口

### 2. 软约束

软约束是默认做法，目的是帮助 agent 高效开工、合理切任务、减少上下文污染。

软约束可以偏离，但需要满足两个条件：

1. 不违反硬约束
2. 在交付说明中说明为什么偏离

适合放进软约束的内容包括：

- 默认任务拆分顺序
- 推荐改动路径
- 默认并行策略
- 推荐测试集
- 当前主要协作面

### 3. 背景说明

背景说明只负责帮助理解，不参与门禁，也不应被实现者当成必须逐字遵守的规则。

适合放进背景说明的内容包括：

- 为什么这样分层
- 为什么这些文件容易膨胀
- 历史演进背景
- 常见失败模式

## 当前文档体系如何对应三档约束

### 以硬约束为主

- `docs/01-系统架构总览.md`
- `docs/02-模块地图.md`
- `docs/03-共享瓶颈点.md`

说明：

- 只有这些文档里的机器可读契约块，才默认进入 contract tests
- 即使在这些文档里，文字说明也不自动等于硬约束，只有契约块才算硬门禁

### 以软约束为主

- `docs/modules/*.md`
- `docs/runbooks/*.md`
- `AGENTS.md`

说明：

- 这些文档主要服务于任务切分、上下文隔离和默认执行路径
- 除非某条规则同时出现在硬约束契约块中，否则按软约束理解

### 以背景说明为主

- 兼容入口文档
- 历史说明
- 运行说明中与架构无关的部分

## 瘦身原则

以后新增架构文档时，先问 3 个问题：

1. 这条规则如果不遵守，会不会造成跨模块破坏？
2. 这条规则是不是已经稳定到值得长期门禁？
3. 这条规则是不是在约束模块之间，而不是模块内部？

只有 3 个问题都回答“是”，才有资格进入硬约束。

否则就降级成软约束或背景说明。
