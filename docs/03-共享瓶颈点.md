# 共享瓶颈点

本文档专门解决并行开发最容易出问题的地方：哪些文件是模块汇合口，为什么它们高冲突，以及应当如何拆任务。

本页中：

- 共享瓶颈点路径列表属于硬约束
- 推荐拆法、协作顺序、场景示例属于软约束
- 为什么这些文件危险属于背景说明

## 什么叫共享瓶颈点

共享瓶颈点指同时满足下面两个条件的文件：

1. 多个模块都会读或改它
2. 它一旦改动，就会影响系统边界、注册关系、装配关系或公共行为

这类文件默认不适合多人同时修改。能并行的，通常不是“同时改这个文件”，而是“一个人改这个文件，其他人改周边模块内部实现”。

## 总体协作规则

1. 触及共享瓶颈点时，任务自动升级为跨模块任务。
2. 共享瓶颈点默认串行，不要同时开两个 agent 去改同一个瓶颈文件。
3. 如果必须并行，先锁定一个 agent 负责瓶颈点接线，其他 agent 只改各自模块内部实现。
4. 共享瓶颈点改动应尽量薄化，只保留装配、注册、转发，不沉积业务逻辑。

## `lib/tool_api.py`

### 为什么是瓶颈点

- GUI 和 agent 共用这个入口
- 它是库存工具的统一表面
- 改动这里容易同时影响调用方、审计链路和测试面

### 允许承载的内容

- 统一入口
- 参数归一化
- 调用转发
- 统一返回格式

### 不应继续堆积的内容

- 大段业务规则实现
- 与单一模块强绑定的特例逻辑

### 推荐拆法

1. 先在 `lib/` 内部实现具体规则或服务
2. 最后再薄改 `lib/tool_api.py` 接线

## `lib/tool_registry.py`

### 为什么是瓶颈点

- 当前它是工具契约、调度绑定、GUI 暴露策略的单一来源
- 改动它会同时影响 agent、GUI、contract tests

### 允许承载的内容

- 描述符定义
- 契约元数据
- 注册与导出逻辑

### 协作要求

- 新增工具时，先明确它属于哪个模块
- 同一批任务里，只允许一个负责人改注册表
- 其他模块围绕描述符结果去适配，不反向发散定义

## `lib/tool_contracts.py`

### 为什么是瓶颈点

- 它对外暴露兼容 facade
- 外部代码和测试会把它当成稳定入口

### 协作要求

- 不要把新的单一来源重新塞回这个文件
- 若注册真相源变化，保持这里尽量薄，只做兼容导出

## `app_gui/tool_bridge.py`

### 为什么是瓶颈点

- 它是 GUI 到核心能力的桥
- 方法签名与桥接策略会同时影响 GUI、Tool API 与测试

### 允许承载的内容

- 路径守卫
- actor context 注入
- 调用绑定
- 失败包装

### 不应承载的内容

- 复杂库存规则
- 具体面板私有逻辑

### 推荐拆法

1. 在 `inventory_core` 或 `gui_application` 里完成真实规则
2. 最后仅在 bridge 中补一层映射

## `app_gui/main.py`

### 为什么是瓶颈点

- 它是 GUI 主窗口装配入口
- 面板 wiring、信号连接、启动流程都会在这里汇合

### 协作要求

- 新功能优先落到 `app_gui/application/` 或 `app_gui/ui/`
- `main.py` 只保留装配、初始化和顶层信号连接
- 若某任务需要大改 `main.py`，优先判断是不是应该先下沉到模块内部

## `app_gui/main_window_flows.py`

### 为什么是瓶颈点

- 这里承接很多跨面板、跨流程的协作逻辑
- 很容易因为“先能跑起来”而不断吸纳杂质

### 协作要求

- 把局部逻辑留在各自模块
- 把业务规则留在核心层
- 这里只做窗口级协调与流程收口

## `app_gui/i18n/translations/en.json`

## `app_gui/i18n/translations/zh-CN.json`

### 为什么是瓶颈点

- 所有 GUI 文案最终都会汇合到这里
- 多人并行时极易产生 key 冲突、遗漏或不同步

### 协作要求

- 文案合并尽量放到最后阶段
- 同一批任务统一命名 key 前缀
- 中英文一起改，不允许只落单边

## 共享瓶颈点与任务拆分建议

### 场景一：新增一个库存工具

推荐拆法：

1. `inventory_core` 实现真实规则
2. 单独一个人改 `lib/tool_registry.py`
3. GUI 或 agent 各自适配调用面

### 场景二：新增一个 GUI 操作流

推荐拆法：

1. `gui_presentation` 先实现面板局部 UI
2. `gui_application` 再接流程编排
3. 最后统一改 `app_gui/main.py` 或 `app_gui/tool_bridge.py`

### 场景三：导入流程接入 agent

推荐拆法：

1. `migration_import` 先定迁移规则
2. `agent_runtime` 再接工具调度
3. 若必须改工具注册，单独收口到 `lib/tool_registry.py`

## 判断标准

如果一个改动满足下面任一条件，就按共享瓶颈点流程处理：

1. 会改公共方法签名
2. 会改工具注册、工具契约或统一返回格式
3. 会改主窗口装配关系
4. 会改跨模块都依赖的翻译 key
5. 会让多个模块同时需要围绕同一个文件适配
