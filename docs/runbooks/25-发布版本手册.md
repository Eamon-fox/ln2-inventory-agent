# 发布版本手册

适用模块：`release_packaging`

本手册描述的是当前正式发布链路，默认前提如下：

- 开发仓库在 MacBook：`~/code/snowfox`
- 网站运行在 Linux 服务器：`ecs:/var/www/snowfox.bio`
- 线上站点实际服务目录：`/var/www/snowfox.bio/dist`
- 安装包与版本元数据发布到 OSS bucket：`snowfox-release`

## 适用场景

优先在以下场景使用本手册：

- 发布一个新的正式版本
- 修正版本元数据并重新发布
- 重新上传安装包、`latest.json` 或 `CHANGELOG.md`
- 在 OSS 上传后同步网站下载页到新版本

如果只是修改安装器脚本、构建脚本或发布文案，而不进行真实发布，可以只读 `docs/modules/15-发布与安装.md`。

## 版本真相源

正式发版时，以下文件必须视为同一批次的同步对象：

1. `app_gui/version.py`
   应用当前版本权威来源。
2. `installer/windows/LN2InventoryAgent.iss`
   Windows 安装器默认版本与产物文件名来源。
3. `latest.json`
   自动更新检查、网站最新版展示和下载入口来源。
4. `CHANGELOG.md`
   版本历史、人类可读发布说明，以及网站历史版本列表来源。
5. `dist/installer/SnowFox-Setup-<version>.exe`
   对外分发的真实安装包。

补充说明：

- `ln2_inventory.spec` 会从 `app_gui/version.py` 读取版本号。
- `CHANGELOG.md` 顶部条目是发布说明唯一真相源。
- `scripts/render_release_artifacts.py` 会从该条目生成 `latest.json.release_notes` 与 `docs/releases/v<version>-github-release.md`。
- 网站“最新版”显示来自网站服务器本地 `latest.json`。
- 网站“历史版本列表”当前来自 OSS 上的 `CHANGELOG.md` 运行时解析。
- 如果版本号只改了其中一部分，默认视为发版未完成。

## 标准发布流程

### 1. 确认发版范围

先确认这次发布属于哪一类：

- 完整新版本：升版本、构建安装包、上传 OSS、同步网站、做验收
- 元数据补发：版本号不变，但要重写 `latest.json`、`CHANGELOG.md`、网站展示或重新上传同名对象
- 仅重传资产：不改版本号，只补传安装包或 OSS 元数据

若当前工作区还有无关改动，先明确这些改动是否属于本次发版内容。不要把其他功能开发和正式发版混在一起。

### 2. 确定版本号与发布日期

建议规则：

- 小修复、链路修正、文案修正：优先 patch
- 有明确新能力但兼容性不变：视范围决定 patch 或 minor
- `CHANGELOG.md` 顶部日期写实际对外发布日期

### 3. 更新版本元数据

至少同步以下文件：

1. `app_gui/version.py`
2. `installer/windows/LN2InventoryAgent.iss`
3. `latest.json`
4. `CHANGELOG.md`

要求：

- `latest.json.download_url` 必须与安装包文件名一致
- `CHANGELOG.md` 顶部条目必须覆盖本次真正要对外说明的内容
- 不要手工分别维护 GitHub Release、自动更新说明和网站最新版说明

### 4. 从 CHANGELOG 生成发布说明产物

在仓库根目录运行：

```bash
python3 scripts/render_release_artifacts.py --version <version>
```

这个脚本会自动生成：

1. `latest.json.release_notes`
2. `docs/releases/v<version>-github-release.md`

因此以下三处说明内容保持同源：

- GitHub Release 页面
- 软件自动更新提示
- 网站下载页“最新版”说明

### 5. 跑版本一致性检查

在仓库根目录运行：

```bash
python3 scripts/validate_version.py
```

通过标准：

- `version.py`
- `latest.json`
- `CHANGELOG.md`
- `installer/windows/LN2InventoryAgent.iss`

以上四处版本一致，且 `latest.json.download_url` 包含正确版本号。

### 6. 构建 Windows 安装包

在 Windows 上执行：

```bat
installer\windows\build_installer.bat
```

至少检查：

- 生成 `dist/installer/SnowFox-Setup-<version>.exe`
- 文件名版本号正确
- 文件时间是本次构建时间
- 文件体积没有明显异常

如果只是元数据补发、安装包本身未变化，可以跳过重构建，但要在交付说明中说明原因。

### 7. 上传 OSS

发布桶：

- `snowfox-release`

必须上传的对象：

1. `dist/installer/SnowFox-Setup-<version>.exe`
2. `latest.json`
3. `CHANGELOG.md`

推荐顺序：

1. 先上传安装包
2. 再上传 `latest.json`
3. 最后上传 `CHANGELOG.md`

原因：

- 如果先更新 `latest.json`，客户端和网站可能先看到新版本，但安装包还不可下载
- 如果不上传 `CHANGELOG.md`，网站历史版本列表会继续显示旧内容

#### 6.1 本机可直接上传时

```bash
python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file dist/installer/SnowFox-Setup-<version>.exe --bucket snowfox-release

python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file latest.json --bucket snowfox-release

python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file CHANGELOG.md --bucket snowfox-release
```

#### 6.2 需要中转上传时

如果当前环境的 OSS 工具不可用，先把文件传到支持 OSS 工具的机器，例如 `mac`：

```bash
scp dist/installer/SnowFox-Setup-<version>.exe mac:~/tmp/
scp latest.json mac:~/tmp/
scp CHANGELOG.md mac:~/tmp/
```

再在远端执行上传脚本：

```bash
ssh mac "python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file ~/tmp/SnowFox-Setup-<version>.exe --bucket snowfox-release"

ssh mac "python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file ~/tmp/latest.json --bucket snowfox-release"

ssh mac "python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file ~/tmp/CHANGELOG.md --bucket snowfox-release"
```

### 8. 验证 OSS 结果

至少验证：

- 安装包对象存在
- `latest.json` 对象存在
- `CHANGELOG.md` 对象存在
- `latest.json.version` 正确
- `latest.json.download_url` 与安装包 URL 一致
- `CHANGELOG.md` 顶部条目是本次版本

如 OSS 开启防盗链，直接 `GET` 或 `HEAD` 返回 `403` 不一定代表上传失败，应改用允许来源的环境或认证工具验证。

### 9. 同步网站

只有在 OSS 上传完成后，才能同步网站。

在 MacBook 开发仓库执行：

```bash
cd ~/code/snowfox
./scripts/sync-website.sh
```

这个脚本会做三件事：

1. 把本地 `latest.json` 通过 SSH 复制到 `ecs:/var/www/snowfox.bio/latest.json`
2. 在 Linux 网站服务器运行 `node scripts/build.js`
3. 校验 `ecs:/var/www/snowfox.bio/dist/latest.json` 的版本与本地一致

注意：

- 网站不会自己因为 OSS 上的 `latest.json` 更新而刷新 `dist/`
- 网站“最新版”依赖服务器本地 `dist/latest.json`
- 所以只上传 OSS、不跑网站同步，下载页仍可能停留在旧版本

### 10. Git 提交、打 Tag、推送

若本次任务包含正式发布提交，再执行：

```bash
git add -A
git commit -m "chore: release v<version>"
git tag -a "v<version>" -m "Release v<version>"
git push
git push --tags
```

注意：

- 若工作区存在未准备发布的其他改动，不要直接盲目 `git add -A`
- 先确认要提交的内容就是本次版本应包含的全部内容

### 11. GitHub Release 正文

建议把 GitHub Release 正文单独落盘到：

- `docs/releases/v<version>-github-release.md`

该文件由 `scripts/render_release_artifacts.py` 自动生成。GitHub 网页发布时直接粘贴该文件内容，不要再手工重写一份不同版本的说明。

### 12. 发布后验收

至少做以下验收：

1. 打开安装包 URL，确认可以下载
2. 读取 OSS 上的 `latest.json`，确认版本号、下载链接、发布说明正确
3. 读取 OSS 上的 `CHANGELOG.md`，确认顶部版本与发布日期正确
4. 打开 `https://snowfox.bio/download.html`，确认最新版展示正确
5. 确认网站历史版本列表包含本次新旧版本信息
6. 如条件允许，在测试机实际运行安装包

## 推荐命令清单

```bash
cd ~/code/snowfox
python3 scripts/render_release_artifacts.py --version <version>
python3 scripts/validate_version.py
./scripts/release.sh
./scripts/sync-website.sh
```

如果不走 `release.sh`，最小手动流程：

```bash
python3 scripts/render_release_artifacts.py --version <version>
python3 scripts/validate_version.py
# Windows 上构建 installer
python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py upload-file dist/installer/SnowFox-Setup-<version>.exe --bucket snowfox-release
python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py upload-file latest.json --bucket snowfox-release
python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py upload-file CHANGELOG.md --bucket snowfox-release
./scripts/sync-website.sh
```

## 常见失败点

### 1. 版本号只改了部分文件

表现：

- 自动更新提示版本不对
- 安装器文件名与 `latest.json` 不一致
- `validate_version.py` 报错

处理：

- 以 `app_gui/version.py` 为起点重新核对四个版本文件

### 2. 网站同步早于 OSS 上传

表现：

- 下载页先显示新版本
- 点击下载返回 404 或旧文件

处理：

1. 先完成安装包、`latest.json`、`CHANGELOG.md` 的 OSS 上传
2. 再重新运行 `./scripts/sync-website.sh`

### 3. 网站历史版本没更新

表现：

- 下载页仍显示旧版本列表

处理：

1. 检查 `CHANGELOG.md` 是否已上传到 `snowfox-release`
2. 检查顶部格式是否为 `## <version> - <date>`

### 4. GitHub Release、自动更新说明、网站最新版说明不一致

表现：

- GitHub Release 正文和自动更新弹窗文案不一致
- 网站下载页最新版说明与应用内更新说明不一致

处理：

```bash
python3 scripts/render_release_artifacts.py --version <version>
```

然后重新上传 `latest.json`，必要时重新同步网站。

### 5. 网站同步脚本报 SSH 或路径错误

表现：

- `ssh ecs` 无法连接
- 脚本提示找不到 `/var/www/snowfox.bio`
- 网站服务器 `build.js` 执行失败

处理：

1. 先单独验证 `ssh ecs`
2. 确认服务器目录 `/var/www/snowfox.bio` 存在
3. 在服务器手动执行 `cd /var/www/snowfox.bio && node scripts/build.js`

## 交付说明建议模板

发版完成后，至少写清楚：

- 本次版本号
- 是否重构建安装包
- 上传了哪些 OSS 对象
- 网站是否已同步
- 验收过哪些入口
- 当前仍未完成的步骤或风险
