# 发布版本手册

适用模块：`release_packaging`

本手册描述当前正式发布链路。当前链路已经收口为：

- 开发仓库在 MacBook：`~/code/snowfox`
- 当前 macOS 实际仓库路径：`/Users/eamon/code/snowfox`
- 当前 Windows 实际仓库路径：`D:\github_repo\ln2-inventory-agent`
- 网站运行在 Linux 服务器：`ecs:/var/www/snowfox.bio`
- 线上站点实际服务目录：`/var/www/snowfox.bio/dist`
- 安装包与版本元数据发布到 OSS bucket：`snowfox-release`
- 正式发布资产包含两个平台：
  - Windows：`SnowFox-Setup-<version>.exe`
  - macOS：`SnowFox-<version>-macOS.pkg`

补充说明：

- Windows 的 `D:\SnowFox` 是安装目录，不是 git 仓库
- 如需在 Windows 上 `git pull` 或重新构建，请进入 `D:\github_repo\ln2-inventory-agent`

## 适用场景

优先在以下场景使用本手册：

- 发布一个新的正式版本
- 修正版本元数据并重新发布
- 重新上传安装包、`latest.json` 或 `CHANGELOG.md`
- 在 OSS 上传后同步网站下载页到新版本
- 创建或补建某个正式版本对应的 GitHub Release

如果只是修改安装器脚本、构建脚本或发布文案，而不进行真实发布，可以只读 `docs/modules/15-发布与安装.md`。

## 版本真相源

正式发版时，以下文件必须视为同一批次的同步对象：

1. `app_gui/version.py`
   应用当前版本权威来源。
2. `installer/windows/LN2InventoryAgent.iss`
   Windows 安装器默认版本与产物文件名来源。
3. `latest.json`
   自动更新检查、网站最新版展示和双平台下载入口来源。
4. `CHANGELOG.md`
   版本历史、人类可读发布说明，以及网站历史版本列表来源。
5. `dist/installer/SnowFox-Setup-<version>.exe`
   对外分发的 Windows 安装包。
6. `dist/installer/SnowFox-<version>-macOS.pkg`
   对外分发的 macOS 安装包。
7. `docs/releases/v<version>-github-release.md`
   GitHub Release 页面正文真相源。
8. GitHub Release assets
   GitHub Release 页面上附带的 Windows/macOS 安装包二进制资产。

补充说明：

- `ln2_inventory.spec` 与 `ln2_inventory.mac.spec` 都会从 `app_gui/version.py` 读取版本号。
- `CHANGELOG.md` 顶部条目是发布说明唯一真相源。
- `scripts/render_release_artifacts.py` 会从该条目生成 `latest.json.release_notes` 与 `docs/releases/v<version>-github-release.md`。
- `docs/releases/v<version>-github-release.md` 是 GitHub Release 正文唯一来源，不应再手工另写一份偏离 `CHANGELOG.md` 的版本说明。
- 网站“最新版”显示来自网站服务器本地 `latest.json`。
- 网站“历史版本列表”当前来自 OSS 上的 `CHANGELOG.md` 运行时解析。
- `latest.json.download_url` 是兼容字段，必须继续指向 Windows 安装包，避免旧客户端更新检查被破坏。

## `latest.json` 规则

正式发布时，`latest.json` 至少必须满足：

```json
{
  "version": "1.3.6",
  "download_url": "https://snowfox-release.oss-cn-beijing.aliyuncs.com/SnowFox-Setup-1.3.6.exe",
  "platforms": {
    "windows": {
      "download_url": "https://snowfox-release.oss-cn-beijing.aliyuncs.com/SnowFox-Setup-1.3.6.exe",
      "asset_name": "SnowFox-Setup-1.3.6.exe",
      "auto_update": true
    },
    "macos": {
      "download_url": "https://snowfox-release.oss-cn-beijing.aliyuncs.com/SnowFox-1.3.6-macOS.pkg",
      "asset_name": "SnowFox-1.3.6-macOS.pkg",
      "auto_update": true
    }
  },
  "release_notes": "..."
}
```

要求：

- 顶层 `download_url` 必须保留，并固定为 Windows `.exe`
- 新版客户端优先读取 `platforms.<platform>.download_url`
- Windows 与 macOS 的 URL、文件名、版本号必须互相一致
- 不要只更新一个平台条目

## 标准发布流程

### 推荐拓扑

正式发版时，当前更推荐采用“单主控机统一收口”的拓扑。

推荐模式：

- 由一台机器承担版本元数据更新、发布说明生成、版本校验、OSS 上传、网站同步、Git 提交与打 tag
- 另一台异平台机器仅在需要时负责产出对应平台安装包
- 是否通过 SSH 到另一台机器执行构建，属于操作层选择，不属于仓库脚本默认职责

建议优先级：

- 在 macOS 上发版时，优先让 macOS 机器作为主控机；需要 Windows 包时，再 SSH 到 Windows 机器构建 `SnowFox-Setup-<version>.exe`
- 在 Windows 上发版时，优先让 Windows 机器作为主控机；需要 macOS 包时，再 SSH 到 macOS 机器构建 `SnowFox-<version>-macOS.pkg`

这样做的目的：

- 把版本号、`latest.json`、`CHANGELOG.md`、OSS 上传、网站同步收口到一个控制面
- 减少“两边都各改了一点，但不是同一批版本”的操作漂移
- 提高发版复现性，同时避免把环境特定细节过度编码进项目脚本

注意：

- 这是当前推荐操作方式，不代表仓库现有脚本已经自动执行跨机 SSH 构建
- 当前 `release.sh` 仍以本机元数据更新和发版检查为主
- SSH、远端路径、远端环境、OSS 上传机器选择，默认都属于操作层
- 如果发版时可用 agent/skill，例如网络访问或 OSS 上传能力，优先在当次操作里临时调用，不要把这些环境细节固化回仓库脚本

### 1. 确认发版范围

先确认这次发布属于哪一类：

- 完整新版本：升版本、构建双平台安装包、上传 OSS、同步网站、做验收
- 完整新版本：升版本、构建双平台安装包、上传 OSS、同步网站、创建 GitHub Release、做验收
- 元数据补发：版本号不变，但要重写 `latest.json`、`CHANGELOG.md`、网站展示或重新上传同名对象
- 仅重传资产：不改版本号，只补传安装包或 OSS 元数据

若当前工作区还有无关改动，先明确这些改动是否属于本次发版内容。不要把其他功能开发和正式发版混在一起。

### 2. 确定版本号与发布日期

建议规则：

- 小修复、链路修正、文案修正：优先 patch
- 有明确新能力但兼容性不变：视范围决定 patch 或 minor
- `CHANGELOG.md` 顶部日期写实际对外发布日期

### 3. 更新版本元数据

至少同步以下文件：

1. `app_gui/version.py`
2. `installer/windows/LN2InventoryAgent.iss`
3. `latest.json`
4. `CHANGELOG.md`

要求：

- `latest.json.download_url` 必须与 Windows 安装包文件名一致
- `latest.json.platforms.windows.asset_name` 必须是 `SnowFox-Setup-<version>.exe`
- `latest.json.platforms.macos.asset_name` 必须是 `SnowFox-<version>-macOS.pkg`
- `CHANGELOG.md` 顶部条目必须覆盖本次真正要对外说明的内容

### 4. 从 CHANGELOG 生成发布说明产物

在仓库根目录运行：

```bash
python3 scripts/render_release_artifacts.py --version <version>
```

这个脚本会自动生成：

1. `latest.json.release_notes`
2. `docs/releases/v<version>-github-release.md`

因此以下三处说明内容保持同源：

- GitHub Release 页面
- 软件自动更新提示
- 网站下载页“最新版”说明

### 5. 跑版本一致性检查

在仓库根目录运行：

```bash
python3 scripts/validate_version.py
```

通过标准：

- `version.py`
- `latest.json`
- `CHANGELOG.md`
- `installer/windows/LN2InventoryAgent.iss`
- `latest.json.download_url`
- `latest.json.platforms.windows`
- `latest.json.platforms.macos`

以上内容全部一致。

### 6. 构建 Windows 安装包

在 Windows 上执行：

```bat
installer\windows\build_installer.bat
```

推荐做法：

- 若当前主控机不是 Windows，可由主控机通过 SSH 到 Windows 机器执行该命令
- 不建议发版时人工分裂成两套独立操作流，最后再靠记忆回填版本状态

至少检查：

- 生成 `dist/installer/SnowFox-Setup-<version>.exe`
- 文件名版本号正确
- 文件时间是本次构建时间
- 文件体积没有明显异常

### 7. 构建 macOS 安装包

在 macOS 上执行：

```bash
bash installer/mac/build_pkg.sh
```

推荐做法：

- 若当前主控机不是 macOS，可由主控机通过 SSH 到 macOS 机器执行该命令
- 构建完成后，再由主控机统一继续后续上传、同步网站和发布动作

至少检查：

- 生成 `dist/installer/SnowFox-<version>-macOS.pkg`
- 文件名版本号正确
- 文件时间是本次构建时间
- 文件体积没有明显异常

说明：

- 当前 macOS 产物默认是未签名 / 未 notarize 的自用包
- 但正式发版元数据已经允许它作为公开下载资产进入同一套发布流

### 8. 上传 OSS

发布桶：

- `snowfox-release`

必须上传的对象：

1. `dist/installer/SnowFox-Setup-<version>.exe`
2. `dist/installer/SnowFox-<version>-macOS.pkg`
3. `latest.json`
4. `CHANGELOG.md`

推荐顺序：

1. 先上传 Windows 安装包
2. 再上传 macOS 安装包
3. 再上传 `latest.json`
4. 最后上传 `CHANGELOG.md`

原因：

- 如果先更新 `latest.json`，客户端和网站可能先看到新版本，但安装包还不可下载
- 如果不上传 `CHANGELOG.md`，网站历史版本列表会继续显示旧内容

#### 8.1 本机可直接上传时

```bash
python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file dist/installer/SnowFox-Setup-<version>.exe --bucket snowfox-release

python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file dist/installer/SnowFox-<version>-macOS.pkg --bucket snowfox-release

python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file latest.json --bucket snowfox-release

python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file CHANGELOG.md --bucket snowfox-release
```

#### 8.2 需要中转上传时

如果当前环境的 OSS 工具不可用，先把文件传到支持 OSS 工具的机器，例如 `mac`：

```bash
scp dist/installer/SnowFox-Setup-<version>.exe mac:~/tmp/
scp dist/installer/SnowFox-<version>-macOS.pkg mac:~/tmp/
scp latest.json mac:~/tmp/
scp CHANGELOG.md mac:~/tmp/
```

再在远端执行上传脚本：

```bash
ssh mac "python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file ~/tmp/SnowFox-Setup-<version>.exe --bucket snowfox-release"

ssh mac "python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file ~/tmp/SnowFox-<version>-macOS.pkg --bucket snowfox-release"

ssh mac "python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file ~/tmp/latest.json --bucket snowfox-release"

ssh mac "python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file ~/tmp/CHANGELOG.md --bucket snowfox-release"
```

边界说明：

- 这里保留的是推荐操作示例，不表示这些上传步骤应该继续被封装进仓库脚本
- 使用哪台机器上传、是否中转、凭证放在哪里，属于操作环境决策
- 这类步骤更适合在实际发版时由人或 agent/skill 执行

### 9. 验证 OSS 结果

至少验证：

- Windows 安装包对象存在
- macOS 安装包对象存在
- `latest.json` 对象存在
- `CHANGELOG.md` 对象存在
- `latest.json.version` 正确
- `latest.json.download_url` 指向 Windows 安装包
- `latest.json.platforms.macos.download_url` 指向 macOS 安装包
- `CHANGELOG.md` 顶部条目是本次版本

如 OSS 开启防盗链，直接 `GET` 或 `HEAD` 返回 `403` 不一定代表上传失败，应改用允许来源的环境或认证工具验证。

### 10. 同步网站

只有在 OSS 上传完成后，才能同步网站。

在 MacBook 开发仓库执行：

```bash
cd ~/code/snowfox
./scripts/sync-website.sh
```

这个脚本会做三件事：

1. 把本地 `latest.json` 通过 SSH 复制到 `ecs:/var/www/snowfox.bio/latest.json`
2. 在 Linux 网站服务器运行 `node scripts/build.js`
3. 校验 `ecs:/var/www/snowfox.bio/dist/latest.json` 的版本与本地一致

注意：

- 网站不会自己因为 OSS 上的 `latest.json` 更新而刷新 `dist/`
- 网站“最新版”依赖服务器本地 `dist/latest.json`
- 所以只上传 OSS、不跑网站同步，下载页仍可能停留在旧版本

### 11. 创建 GitHub Release

完整发版默认必须创建对应版本的 GitHub Release。

要求：

- 使用 `docs/releases/v<version>-github-release.md` 作为唯一正文来源
- 不要在 GitHub 页面里手工改写另一版说明
- Release tag 必须与本次正式版本一致
- 完整发版默认必须把 Windows 与 macOS 安装包一起上传到该 Release
- 若 Release 已存在，补发时应更新现有 Release，而不是新造不同 tag

推荐在仓库根目录执行：

```bash
gh release create "v<version>" \
  --title "SnowFox v<version>" \
  --notes-file "docs/releases/v<version>-github-release.md" \
  --verify-tag

gh release upload "v<version>" \
  "dist/installer/SnowFox-Setup-<version>.exe" \
  "dist/installer/SnowFox-<version>-macOS.pkg"
```

如果该版本的 Release 已存在，改用：

```bash
gh release edit "v<version>" \
  --title "SnowFox v<version>" \
  --notes-file "docs/releases/v<version>-github-release.md"

gh release upload "v<version>" \
  "dist/installer/SnowFox-Setup-<version>.exe" \
  "dist/installer/SnowFox-<version>-macOS.pkg" \
  --clobber
```

### 12. Git 提交、打 Tag、推送

若本次任务包含正式发布提交，再执行：

```bash
git add -A
git commit -m "chore: release v<version>"
git tag -a "v<version>" -m "Release v<version>"
git push
git push --tags
```

注意：

- 若工作区存在未准备发布的其他改动，不要直接盲目 `git add -A`
- 先确认要提交的内容就是本次版本应包含的全部内容

### 13. GitHub Release 正文

必须把 GitHub Release 正文单独落盘到：

- `docs/releases/v<version>-github-release.md`

该文件由 `scripts/render_release_artifacts.py` 自动生成。无论使用 GitHub 网页还是 `gh` CLI，都必须直接使用这个文件，不要再手工重写一份不同版本的说明。

### 14. 发布后验收

至少做以下验收：

1. 打开 Windows 安装包 URL，确认可以下载
2. 打开 macOS 安装包 URL，确认可以下载
3. 读取 OSS 上的 `latest.json`，确认版本号、双平台下载链接、发布说明正确
4. 打开网站下载页，确认最新版已刷新到目标版本
5. 打开 GitHub Release 页面，确认标题、tag、正文与版本一致
6. 确认 GitHub Release 页面上同时存在 Windows `.exe` 和 macOS `.pkg` 资产

## 常见失败模式

- 只更新 `latest.json.download_url`，忘了更新 `platforms.macos`
- 只上传 Windows 安装包，没有上传 macOS 安装包
- 先同步网站，后上传 OSS，导致下载页先显示新版本但资产不可下载
- 手工改了 GitHub Release 文本，没有重新从 `CHANGELOG.md` 生成
- 完整发版做完了 tag 和 push，但没有创建 GitHub Release
- GitHub Release 创建了正文，但忘了上传安装包资产
- 安装包文件名与 `latest.json` 不一致
