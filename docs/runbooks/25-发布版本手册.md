# 发布版本手册

适用模块：`release_packaging`

本手册按软约束理解，目标是把一次正式发版拆成稳定、可复用、可核查的标准流程。

## 适用场景

在以下场景优先使用本手册：

- 正式发布一个新版本
- 修正版本元数据后重新发布
- 重传安装包或 `latest.json`
- 同步网站下载入口和历史版本列表

如果只是修改构建脚本、安装器脚本或发布文案，而不做真实发布，可以只读 `docs/modules/15-发布与安装.md`。

## 版本真相源

正式发版时，以下文件必须视为同一批次的同步对象：

1. `app_gui/version.py`
   当前应用版本权威来源。
2. `installer/windows/LN2InventoryAgent.iss`
   Windows 安装器默认版本与输出文件名来源。
3. `latest.json`
   自动更新检查和 OSS 下载入口来源。
4. `CHANGELOG.md`
   版本历史、人类可读发布说明，以及网站历史版本 list 的内容来源。
5. `dist/installer/SnowFox-Setup-<version>.exe`
   真实对外分发的 Windows 安装包。

补充说明：

- `ln2_inventory.spec` 会从 `app_gui/version.py` 读取版本号。
- 网站历史版本 list 由 `scripts/update_history_versions.py` 从 `CHANGELOG.md` 生成。
- 如果版本号只改了其中一部分，默认视为未完成发版。

## 标准发布流程

### 1. 确认发版范围

先确认这次发布属于什么类型：

- 完整新版本：需要升版本、构建、上传、同步网站、做验收
- 元数据补发：版本号不变，但要重写 `CHANGELOG.md`、`latest.json`、网站文案或重新上传相同文件名资产
- 仅重传资产：不改版本号，只补传 OSS 文件并验证 URL

若当前工作区还有其他模块未完成改动，先明确这些改动是否属于本次发版内容。不要把无关脏改动和版本发布混在一起。

### 2. 确定版本号与发布日期

规则建议：

- 小修复、文案修正、发布链路修正：优先 patch 版本
- 有明确新能力但兼容性不变：视范围决定 patch 或 minor
- `CHANGELOG.md` 顶部条目的日期应写实际对外发布日期

### 3. 更新版本与发布说明

至少同步以下文件：

1. `app_gui/version.py`
2. `installer/windows/LN2InventoryAgent.iss`
3. `latest.json`
4. `CHANGELOG.md`

要求：

- `latest.json.download_url` 必须与安装包文件名一致
- `latest.json.release_notes` 面向最终用户，建议写成多行列表
- `CHANGELOG.md` 面向仓库和网站，可写得比 `latest.json` 更详细
- 如果网站历史版本 list 需要更多信息，应优先从 `CHANGELOG.md` 派生，而不是手填多个地方

### 4. 跑版本一致性检查

在仓库根目录运行：

```bash
python scripts/validate_version.py
```

通过标准：

- `version.py`
- `latest.json`
- `CHANGELOG.md`
- `installer/windows/LN2InventoryAgent.iss`

以上四处版本一致，且 `latest.json.download_url` 包含正确版本号。

### 5. 构建 Windows 安装包

在 Windows 上执行：

```bash
installer\windows\build_installer.bat
```

最少检查以下结果：

- 生成 `dist/installer/SnowFox-Setup-<version>.exe`
- 文件名版本号正确
- 文件时间为本次构建时间
- 文件体积与预期量级一致，没有出现明显异常

如果只做元数据补发、安装包本身未变化，可以跳过重构建，但要在交付说明中写清楚原因。

### 6. 上传 OSS

默认发布桶：

- `snowfox-release`

默认要上传的对象：

1. `dist/installer/SnowFox-Setup-<version>.exe`
2. `latest.json`

推荐顺序：

1. 先上传安装包
2. 再上传 `latest.json`

原因：

- 如果先更新 `latest.json`，客户端可能先看到新版本，但 OSS 上的安装包还没准备好

#### 6.1 本机可直接上传时

如果当前机器可直接运行 OSS 工具：

```bash
python ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file dist/installer/SnowFox-Setup-<version>.exe \
  --bucket snowfox-release

python ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file latest.json \
  --bucket snowfox-release
```

#### 6.2 Windows 本机无法直接上传时

如果当前 Windows 环境的 OSS 工具链不可用，采用“中转到支持环境再上传”的流程：

1. 用 `scp` 把安装包和 `latest.json` 传到 `mac` 或其他已配置好 OSS 工具的机器
2. 在远端执行 OSS 上传脚本
3. 记录最终公开 URL

示例：

```bash
scp dist/installer/SnowFox-Setup-<version>.exe mac:~/tmp/
scp latest.json mac:~/tmp/

ssh mac "python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file ~/tmp/SnowFox-Setup-<version>.exe --bucket snowfox-release"

ssh mac "python3 ~/.agents/skills/aliyun-oss-upload/scripts/oss_upload.py \
  upload-file ~/tmp/latest.json --bucket snowfox-release"
```

### 7. 验证 OSS 结果

至少验证以下内容：

- 安装包对象存在
- `latest.json` 对象存在
- 安装包公开 URL 与 `latest.json.download_url` 一致
- 通过允许来源或带认证的方式验证 `latest.json.version` 正确
- `release_notes` 是本次版本说明，而不是旧文案

建议额外记录：

- 安装包公开 URL
- `latest.json` 公开 URL
- 安装包文件大小

如果 OSS 开启了防盗链：

- 来自未授权来源的直接 GET/HEAD 可能返回 `403`
- 这不一定代表上传失败
- 应改用 OSS 认证工具、允许来源的页面环境，或受策略允许的客户端来做验证

### 8. 同步网站下载入口与历史版本 list

如果当前环境具备网站工作副本和发布权限，再执行：

```bash
python scripts/sync_website_version.py
python scripts/update_history_versions.py
```

职责分工：

- `sync_website_version.py`
  同步最新版下载链接与版本号展示
- `update_history_versions.py`
  从 `CHANGELOG.md` 生成历史版本 list

当前约定：

- 历史版本 list 至少显示版本号和发布日期
- 悬停提示可显示从 changelog 提取的版本摘要

### 9. Git 提交、打 Tag、推送

如果本次任务包含真正的版本发布提交，再执行：

```bash
git add -A
git commit -m "chore: release v<version>"
git tag -a "v<version>" -m "Release v<version>"
git push
git push --tags
```

注意：

- 若工作区存在未准备发布的其他改动，不要直接 `git add -A`
- 先确认这批代码就是本次版本应包含的全部内容

### 9.1 GitHub Release 正文

建议把每个正式版本的 GitHub Release 正文单独落盘到：

- `docs/releases/v<version>-github-release.md`

这样可以避免只在 GitHub 网页里临时编辑，导致仓库内没有留档。

### 10. 发布后验收

至少做以下验收：

1. 打开安装包 URL，确认可以下载
2. 在受防盗链策略允许的来源中访问 `latest.json`，确认版本号、下载链接、发布说明正确
3. 如果网站已同步，确认首页下载按钮和下载页最新版链接正确
4. 如果历史版本 list 已同步，确认显示版本号、日期与链接正确
5. 如条件允许，在一台测试机上实际运行安装包

## 常见失败点

### 1. 版本号只改了部分文件

表现：

- 自动更新弹窗版本不对
- 安装器文件名与 `latest.json` 不一致
- `validate_version.py` 报错

处理：

- 以 `app_gui/version.py` 为起点重新核对四个版本文件

### 2. 先上传 `latest.json` 导致客户端提前看到新版本

表现：

- 更新弹窗出现了
- 下载链接返回 404 或旧文件

处理：

- 重新按“安装包先、`latest.json` 后”的顺序上传

### 3. 网站历史版本 list 没更新

表现：

- 下载页仍显示旧版本列表

处理：

- 检查 `CHANGELOG.md` 顶部格式是否为 `## <version> - <date>`
- 检查网站工作副本是否存在 `download.html`
- 重新运行 `python scripts/update_history_versions.py`

### 4. Windows 上 OSS 工具不可用

表现：

- 本机无法直接执行 OSS 上传脚本

处理：

- 走 SSH/SCP 中转流程，把文件传到 `mac` 或其他支持环境后再上传

## 交付说明建议模板

发版完成后，建议在交付说明中至少写清楚：

- 本次版本号
- 是否重构建安装包
- 上传了哪些 OSS 对象
- 网站是否已同步
- 跑了哪些校验命令
- 当前仍未完成的步骤或风险
