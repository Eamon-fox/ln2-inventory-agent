# 修改导入迁移前必读

适用模块：

- `migration_import`

当一个需求主要影响导入工作区、预转换、迁移模板、候选 YAML 验收、导入确认流或导入后 handoff 时，先读本文档。

本文属于软约束，描述默认执行路径。必要时可以偏离，但不得违反硬约束。

## 先判断影响范围

以下改动通常属于纯导入迁移任务：

- 改 `migrate/` 工作区协议
- 改 `migration_assets/` 模板
- 改 `lib/import_*`
- 改 `lib/migrate_*`
- 改 `app_gui/import_journey.py`

以下改动会升级为跨模块：

- 改最终库存模型
- 改统一工具契约
- 改主窗口接收导入结果的方式
- 改 agent 的工具暴露面

## 开工顺序

1. 先读 `docs/modules/14-导入迁移.md`
2. 保证所有中间产物仍限定在 `migrate/` 工作区
3. 先改预处理、模板或验收规则
4. 最后再改导入完成后的 handoff
5. 如需接 agent，最后才收口到共享注册或确认流程

## 工作区边界

默认工作范围：

- `migrate/inputs/`
- `migrate/normalized/`
- `migrate/output/`

除非用户明确要求，不要在导入流程里直接读取或写入现有 `inventories/` 数据集做比较或修补。

## 明确禁止

1. 不要跳过候选 YAML 验收直接导入。
2. 不要把迁移细节塞进 GUI 展示层。
3. 不要把迁移真相藏在模板文案里而不写进代码与约束。
4. 不要混用 `migrate/` 工作区和正式库存目录。
5. 不要在导入模块里定义独立库存真相源。

## 常见拆分法

### 改预处理与标准化

1. 先改 `migrate/` 或预转换逻辑
2. 再改模板与清单
3. 最后改验收与 handoff

### 改导入确认流

1. 先改导入验收与阻塞条件
2. 再改 GUI 或 agent 的确认流
3. 最后校对目标数据集切换语义

## 最小测试集

- `pytest -q tests/integration/migration`

如果动到导入完成后的 GUI handoff：

- `pytest -q tests/integration/migration tests/integration/gui`

## 完成标准

1. 导入工作区边界仍然清楚
2. 外部输入的脏数据处理仍留在迁移模块
3. 最终库存真相仍由核心层定义
4. 导入确认、验收、handoff 三者语义一致
