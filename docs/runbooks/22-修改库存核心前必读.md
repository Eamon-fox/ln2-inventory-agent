# 修改库存核心前必读

适用模块：

- `inventory_core`

当一个需求会改变 Tool API、库存规则、字段规则、工具注册、统一返回格式、审计链路或领域事件时，先读本文档。

本文属于软约束，描述默认执行路径。必要时可以偏离，但不得违反硬约束。

## 先判断是不是高风险核心改动

以下改动都属于高风险：

- 改 `lib/tool_api.py`
- 改 `lib/tool_registry.py`
- 改 `lib/domain/events.py`
- 改统一写入链路、审计链路或校验逻辑

这些改动默认不适合多人同时并行。

## 开工顺序

1. 先读 `docs/modules/13-库存核心.md`
2. 判断本次是“规则实现变更”还是“统一入口接线变更”
3. 优先把规则实现放在更内层模块或服务函数
4. 最后薄改 `lib/tool_api.py`
5. 若必须新增工具，最后一步再改 `lib/tool_registry.py`
6. 历史 YAML 兼容统一收口到升级入口，不要把 legacy 分支继续散落到运行期

## 推荐拆分法

### 改业务规则

1. 先改具体规则实现
2. 再改统一入口接线
3. 最后补测试和契约检查

### 新增工具

1. 先定义工具属于哪个模块
2. 先完成真实规则实现
3. 单独收口到 `lib/tool_registry.py`
4. 再让 agent / GUI 各自适配

## 明确禁止

1. 不要把 GUI 私有需求或 agent 私有行为直接写死在核心层。
2. 不要在多个文件重复维护同一份工具契约。
3. 不要让 `tool_api.py` 继续堆成巨型实现文件。
4. 不要把历史文件兼容重新做成 steady-state runtime fallback。
5. 不要引入 `PySide6` 或 GUI 状态对象。

## 高冲突文件

- `lib/tool_api.py`
- `lib/tool_registry.py`

建议协作方式：

1. 一个人改共享入口
2. 其他人改规则实现或调用侧
3. 最后合并契约与接线

## 最小测试集

- `pytest -q tests/integration/inventory`
- `pytest -q tests/unit`

如果改了工具契约、架构边界或领域入口：

- `pytest -q tests/contract/test_tool_contracts_single_source.py`
- `pytest -q tests/contract/test_architecture_dependencies.py`

## 完成标准

1. 规则真相只保留一份
2. 共享入口保持薄
3. 没有把 UI/Agent 特例反向注入核心层
4. 契约、入口、测试和文档保持一致
