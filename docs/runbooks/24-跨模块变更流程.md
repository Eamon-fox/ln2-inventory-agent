# 跨模块变更流程

当一个需求同时影响两个或以上模块，或者命中共享瓶颈点时，按本文档执行。

本文属于软约束，描述默认的跨模块收口方式。必要时可以偏离，但不得违反硬约束。

## 什么时候算跨模块

满足下面任一条件，就按跨模块处理：

1. 需要同时改两个模块的拥有路径
2. 需要改共享瓶颈点
3. 需要新增跨模块依赖
4. 需要新增稳定公共入口
5. 一个模块的改动会强制另一个模块同步适配

## 第一步：先拆成两类改动

所有跨模块需求，优先拆成下面两层：

1. **共享入口 / 契约变更**
2. **模块内部实现变更**

这是默认拆法，不要直接把两类改动混成一个大 patch。

## 第二步：先锁定共享瓶颈点负责人

共享瓶颈点包括但不限于：

- `lib/tool_api.py`
- `lib/tool_registry.py`
- `app_gui/tool_bridge.py`
- `app_gui/main.py`
- `app_gui/main_window_flows.py`
- `app_gui/i18n/translations/en.json`
- `app_gui/i18n/translations/zh-CN.json`

规则：

1. 同一时间只允许一个负责人改某个共享瓶颈文件
2. 其他并行任务只能改模块内部实现
3. 最后统一合并到共享入口

## 第三步：文档先行

如果本次跨模块变更会影响以下任一内容，先改文档再改代码：

- 模块边界
- 依赖方向
- 稳定入口
- 共享瓶颈点列表
- 任务切分方式

需要优先更新的文档通常是：

- `docs/01-系统架构总览.md`
- `docs/02-模块地图.md`
- `docs/03-共享瓶颈点.md`
- 对应的 `docs/modules/*.md`
- 对应的 `docs/runbooks/*.md`

## 推荐执行顺序

1. 定模块归属
2. 定共享瓶颈点负责人
3. 先更新文档
4. 各模块并行改内部实现
5. 最后改共享入口或桥接
6. 最后统一翻译、文案、测试与回归说明

## 常见跨模块场景

### 场景一：新增一个库存工具，同时给 GUI 和 Agent 用

推荐顺序：

1. `inventory_core` 完成真实规则
2. 单独收口到 `lib/tool_registry.py`，补齐契约、分类和显式写入口
3. `agent_runtime` 在自己的运行时注册表里适配调度、提示、hooks 与 status
4. `gui_application` / `gui_presentation` 适配入口与展示

不要在第 2 步顺手把 agent 的提示、hooks、状态文案也塞进共享注册表。

### 场景二：新增一个 GUI 流程，同时需要 Agent 产出计划项

推荐顺序：

1. 先定共享数据结构或稳定入口
2. Agent 侧和 GUI 侧分别改内部实现
3. 最后在 bridge、main flow 或主窗口层做收口

### 场景四：新增本地开放 API

推荐顺序：

1. 先在文档里锁定硬边界，只写 loopback / read-only / managed-session-switch / GUI handoff / no-agent-runtime 这类规则
2. `inventory_core` 保持读/校验真相源不变；数据集切换继续复用现有受管路径规则
3. `gui_application` 实现 transport、route allowlist、主线程 handoff、受管数据集切换与计划暂存收口
4. 最后再在 `main.py` 接服务启停与设置项

不要把端口号、HTTP 框架、handler 命名这类实现细节写进硬契约。

### 场景三：修改导入流程，同时影响最终库存模型

推荐顺序：

1. 先由 `inventory_core` 明确最终模型
2. `migration_import` 适配预处理与验收
3. 如有 agent / GUI 交互变化，再分别适配

## 明确禁止

1. 不要把跨模块需求假装成“顺手小改”直接改共享入口。
2. 不要多人同时改同一个共享瓶颈点文件。
3. 不要先改 UI 或 prompt，再倒逼核心层补契约。
4. 不要在没有文档更新的情况下调整稳定入口。

## 最小测试策略

跨模块任务至少满足：

1. 每个受影响模块跑自己的最小测试集
2. 共享瓶颈点相关 contract tests 要跑
3. 说明哪些部分这次没有覆盖到

常见最小组合：

- `pytest -q tests/integration/gui tests/integration/plan`
- `pytest -q tests/integration/agent`
- `pytest -q tests/integration/inventory`
- `pytest -q tests/integration/migration`
- `pytest -q tests/contract`

## 完成标准

1. 每个改动都能归属到明确模块
2. 共享瓶颈点改动已收口
3. 文档、契约、代码与测试语义一致
4. 交付说明中明确写出：
   - 本次涉及哪些模块
   - 哪些共享瓶颈点被改动
   - 哪些改动是串行收口的
   - 哪些测试已执行
