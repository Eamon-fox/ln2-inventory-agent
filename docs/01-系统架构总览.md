# 系统架构总览

本文档回答 6 个问题：系统是什么、怎么分层、怎么切模块、依赖怎么走、哪些地方是共享瓶颈点、接到任务后应该如何安全变更。

本页采用“硬约束 + 软约束 + 背景说明”三档模型：

- 只有文末机器可读契约块属于硬约束
- 其余文字说明以软约束和背景说明为主
- 如需门禁，请优先把规则收缩到契约块，而不是扩大文字性规定

## 1. 系统定义

SnowFox 当前本质上是一个面向桌面端的分层单体应用：GUI 与 AI 智能体是两个上层入口，两者共享统一的 `lib/tool_api.py` 业务内核，底层数据以 YAML 数据集为主存储，所有写操作都要经过校验、备份与审计链路。

在此基础上，允许存在一个嵌入 GUI 进程的本地开放 API 入口，但它只用于 loopback 场景下的只读查询/校验、受管数据集会话切换与 GUI handoff，不承担远程服务职责，也不直接暴露 agent 运行时能力。

对历史数据集的兼容策略已经收口为“首开自动迁移”：当系统首次打开仍包含 legacy 字段或旧结构别名的 YAML 时，先创建备份，再原地 canonicalize，并记录审计事件；迁移完成后，日常运行链路只面向 canonical 数据模型。

这意味着它不是微服务系统，也不是前后端分离系统。后续架构演进的重点，不是把系统拆得更碎，而是把边界写得更清楚，让人和 agent 都能安全修改。

## 2. 分层结构

当前架构按 4 层理解：

1. **展示层**
   对应 `app_gui/ui/`。
   负责面板、对话框、表格、交互控件与视觉反馈。

2. **应用协调层**
   对应 `app_gui/application/`、`app_gui/main.py`、`app_gui/main_window_flows.py`、`app_gui/plan_*.py`、`app_gui/tool_bridge.py`。
   负责流程编排、事件分发、计划执行、主窗口装配以及 GUI 到核心能力的桥接。

3. **核心业务层**
   对应 `agent/` 与 `lib/` 的业务能力部分。
   负责工具调度、统一 Tool API、库存规则、工具契约、导入规则与业务级校验。

4. **基础设施层**
   对应 `lib/` 内的 YAML I/O、配置、路径策略、备份、审计与外部环境适配能力。

依赖方向必须保持为：

`展示层 -> 应用协调层 -> 核心业务层 -> 基础设施层`

## 3. 模块切片

为了支持并行开发，当前仓库按 6 个稳定模块切片理解。

这里的模块切片是当前推荐骨架，主要用于任务归类和并行切分，默认按软约束理解。真正的 ownership 以 `02-模块地图.md` 的机器可读契约为准。

1. `gui_presentation`
   面向用户界面的展示与交互组件。

2. `gui_application`
   面向 GUI 的流程编排、状态协调与跨面板协作。

3. `agent_runtime`
   面向 AI 智能体的推理循环、工具调度与工具输入校验。

4. `inventory_core`
   面向库存领域的统一工具入口、规则、校验、注册表与数据处理。

5. `migration_import`
   面向导入、迁移、验收与迁移模板。

6. `release_packaging`
   面向安装、打包、发布脚本与版本文件。

详细 ownership、稳定入口、依赖规则与测试责任见 `02-模块地图.md`。

## 4. 依赖规则

以下规则是当前架构默认原则。只有和层级依赖直接相关的部分进入硬约束；其余作为软约束帮助 agent 做出更稳妥的实现选择：

1. 展示层只负责展示与输入采集，不负责定义库存写入规则。
2. GUI 流程编排必须优先通过应用协调层完成，不要把跨面板协作塞回单个 UI 组件。
3. GUI 与 agent 都应共享同一套库存规则，不允许在 `agent/` 中复制一套独立业务逻辑。
4. 核心业务层不得依赖 `PySide6` 或具体窗口/控件对象。
5. 发布与安装模块不得承载库存规则、校验逻辑或数据迁移策略。
6. 新增跨模块依赖时，先更新文档，再允许改代码。

## 5. 共享瓶颈点

当前系统存在一批天然的共享入口文件，它们不是某个模块的纯内部实现，而是多个模块汇合的接口面。共享瓶颈点列表本身是硬约束；为什么它们危险、建议如何拆任务，属于软约束。

- `lib/tool_api.py`
- `lib/tool_registry.py`
- `app_gui/tool_bridge.py`
- `app_gui/main.py`
- `app_gui/main_window_flows.py`
- `app_gui/i18n/translations/en.json`
- `app_gui/i18n/translations/zh-CN.json`

详细原因与协作策略见 `03-共享瓶颈点.md`。

## 6. 变更规则

接到任何需求时，按下面流程处理：

1. 先判断它属于哪个模块切片。
2. 如果只属于一个模块，只在该模块拥有路径内优先修改。
3. 如果命中共享瓶颈点，视为跨模块任务，默认串行处理。
4. 如果需要新增稳定入口、调整依赖方向或新增跨模块调用，先更新文档。
5. 如果一个任务同时需要多个模块改动，优先把“共享入口改动”和“模块内部实现改动”拆成两个阶段。
6. 每次交付都要说明本次改动归属哪个模块、是否触及共享瓶颈点、跑了哪些测试。

## 附：当前稳定总入口

为了帮助 agent 快速定位，当前有 4 个最重要的系统总入口：

1. `lib/tool_api.py`
   统一库存工具入口。

   工具契约的唯一真相源在 `lib/tool_registry.py`；`tool_api.py` 只承载统一入口与调用转发。

   写工具的跨模块解析也必须以 `lib/tool_registry.py` 中显式声明的入口为准，不要在 GUI/Plan/Agent 侧再通过 `tool_` 前缀拼接猜测底层函数名。

2. `app_gui/application/use_cases.py`
   GUI 应用协调层的用例入口。

3. `agent/tool_runner.py`
   智能体工具调度入口。

   tool hooks、status 文案、输入补充规则与错误提示等运行时特例，归 `agent/` 内部运行时注册表所有，不应继续回流到 `lib/tool_registry.py` 这类共享瓶颈点。

4. `app_gui/main.py`
   GUI 装配与主窗口入口。
